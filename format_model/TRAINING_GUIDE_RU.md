# Инструкция для новичка: обучение проверки форматирования

Этот файл объясняет очень простыми шагами, как "обучать" проверку форматирования на ваших примерах.

## 1) Что уже сделано в проекте

- В `bot.py` добавлен RAG-этап проверки перед Gemini.
- Примеры лежат в `format_model/examples.jsonl`.
- Скрипт `format_model/prepare_training_data.py` проверяет примеры и собирает train/valid файлы.

Важно: в этом проекте "обучение" работает по схеме RAG:
- вы добавляете пары `плохой -> хороший`,
- бот перед проверкой выбирает похожие пары,
- и просит модель оформить текст по этим эталонам.

## 2) В каком виде вставлять примеры

Используйте только текстовый формат **JSONL** (не скриншоты).

Одна строка = один JSON-объект:

```json
{"bad":"...плохой текст...","good":"...хороший текст...","tag":"interview_qa","notes":"краткий комментарий"}
```

Поля:
- `bad` — проблемная верстка (как бывает "до исправления").
- `good` — эталонная верстка (как должно быть в Telegram).
- `tag` — тип случая (например: `interview_qa`, `stats_list`, `speaker_quote`).
- `notes` — коротко, что именно исправили.

## 3) Правила подготовки пары `bad -> good`

1. Смысл текста должен быть одинаковым в `bad` и `good`.
2. Меняем только верстку:
   - переносы строк,
   - пустые строки,
   - теги `<b>` и `<blockquote>`,
   - оформление кавычек `«...»`,
   - позицию эмодзи/хештега.
3. Не добавляйте ссылки и лишние HTML-теги.
4. Перед хештегом ставьте пустую строку, если так принято в канале.
5. Для интервью используйте шаблон:
   - `❓ <b>Вопрос?</b>`
   - пустая строка
   - `🎙️ <b>Имя Фамилия</b>:`
   - `<blockquote>«Ответ»</blockquote>`

## 4) Как добавить новые примеры (пошагово)

1. Откройте файл `format_model/examples.jsonl`.
2. Перейдите в конец файла.
3. Добавьте новую строку в формате JSON (см. пункт 2).
4. Убедитесь, что каждая запись находится на одной строке.
5. Сохраните файл.

Совет: добавляйте примеры, где бот ошибся на реальных постах.

## 5) Как "запустить обучение"

В этом проекте обучение = обновление набора примеров и его валидация.

Запуск:

```bash
python format_model/prepare_training_data.py
```

Результат:
- `format_model/build/train.jsonl`
- `format_model/build/valid.jsonl`

Если скрипт ругается на строку, откройте указанную строку в `examples.jsonl` и исправьте JSON.

## 6) Как работает в GitHub Actions

При запуске workflow (`step1.yml`, `step2.yml`, `step3.yml`):
- бот читает `format_model/examples.jsonl`,
- выбирает похожие примеры,
- делает RAG-проверку форматирования,
- если RAG дал плохой ответ, используется безопасный fallback.

Нужные переменные:
- `OPENROUTER_KEY`
- `OPENROUTER_MODEL`
- `FORMAT_RAG_ENABLED=1`

Опционально:
- `FORMAT_EXAMPLES_PATH` (по умолчанию `format_model/examples.jsonl`)
- `FORMAT_RAG_TOP_K` (по умолчанию `4`)
- `FORMAT_RAG_MAX_EXAMPLE_TEXT` (по умолчанию `1200`)
- `FORMAT_RAG_TIMEOUT` (по умолчанию `18`)

## 7) Минимальный план обслуживания (раз в неделю)

1. Добавить 10-30 новых проблемных случаев в `examples.jsonl`.
2. Запустить `python format_model/prepare_training_data.py`.
3. Закоммитить изменения в репозиторий.
4. Проверить 1-2 запуска GitHub Actions.

Чем больше качественных пар `bad -> good`, тем лучше и стабильнее становится проверка.
