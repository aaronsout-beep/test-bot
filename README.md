# NicoPaz / Kendry Paez Telegram Bot

Этот бот ищет свежие посты с медиа, переводит текст на русский, аккуратно
правит имена/термины, приводит пост к Telegram-формату и отправляет в канал.

Инструкция написана максимально простым языком. Если вы не программировали
раньше, просто идите сверху вниз и не пытайтесь сразу понять весь файл `bot.py`.

## Самая короткая версия

Чтобы бот работал, нужно:

1. Положить код в GitHub-репозиторий.
2. В GitHub открыть `Settings` -> `Secrets and variables` -> `Actions`.
3. Добавить секреты: `BOT_TOKEN`, `CHANNEL_ID`, `DEEPL_KEY`.
4. Для нормальной редактуры добавить `GEMINI_API_KEY`.
5. Для RAG-проверки верстки и AI-проверки дублей добавить `OPENROUTER_KEY`.
6. В `Actions` запустить нужный workflow.

Если нет `BOT_TOKEN`, `CHANNEL_ID` или `DEEPL_KEY`, бот не запустится.

## Какие файлы важны

`bot.py` - главный файл. Его запускает GitHub Actions.

`glossary.py` - словарь плохих и хороших написаний. Здесь исправляются имена,
фамилии и футбольные термины.

`config.env.example` - пример настроек. Это не настоящие секреты, а шпаргалка,
как должны называться переменные.

`.github/workflows/` - расписание и команды запуска в GitHub Actions.

`published_ids.json` - кэш уже опубликованных постов. GitHub Actions сохраняет
его через cache, чтобы бот не публиковал одно и то же.

`source_news_cache.json` - кэш смысловых дублей.

`format_model/examples.jsonl` - примеры хорошей верстки для RAG-проверки.

`format_model/hard_cases.jsonl` - сюда попадают случаи, когда AI предложил
опасную или неправильную правку.

## Где настраивать секреты в GitHub

Откройте репозиторий на GitHub.

1. Нажмите `Settings`.
2. В левом меню нажмите `Secrets and variables`.
3. Нажмите `Actions`.
4. Нажмите `New repository secret`.
5. В поле `Name` пишите название переменной, например `BOT_TOKEN`.
6. В поле `Secret` вставьте значение.
7. Нажмите `Add secret`.

Важно: в GitHub Secrets не надо писать кавычки. Просто вставляйте значение.

Правильно:

```text
BOT_TOKEN
7123456789:AAF_xxxxxxxxxxxxxxxxxxxxxxx
```

Неправильно:

```text
BOT_TOKEN="7123456789:AAF_xxxxxxxxxxxxxxxxxxxxxxx"
```

## Обязательные настройки

### BOT_TOKEN

Токен Telegram-бота.

Где взять:

1. Откройте Telegram.
2. Найдите `@BotFather`.
3. Напишите `/newbot`.
4. Следуйте подсказкам.
5. BotFather выдаст строку вида `7123456789:AAF...`.

Эту строку добавьте в GitHub secret `BOT_TOKEN`.

### CHANNEL_ID

ID Telegram-канала, куда бот публикует посты.

Обычно выглядит так:

```text
-1001234567890
```

Важно:

1. Бот должен быть администратором канала.
2. У бота должно быть право публиковать сообщения.
3. `CHANNEL_ID` почти всегда начинается с `-100`.

### DEEPL_KEY

Ключ DeepL для перевода на русский.

Где взять:

1. Зарегистрируйтесь в DeepL API.
2. Откройте личный кабинет.
3. Скопируйте API key.
4. Добавьте его в GitHub secret `DEEPL_KEY`.

DeepL делает первый перевод. Gemini не переводит текст с нуля, он только
проверяет и редактирует уже готовый русский текст.

## Настройки Gemini

### GEMINI_API_KEY

Ключ Google Gemini.

Он нужен для трех разных вещей:

1. Строгая проверка глоссария.
2. Литературная редактура русского текста.
3. Проверка Telegram-верстки.

Если `GEMINI_API_KEY` не задан, бот всё равно будет переводить через DeepL, но:

1. глоссарий будет исправляться только локальными правилами;
2. литературная редактура Gemini будет пропущена;
3. Gemini-проверка верстки будет пропущена.

### LAYOUT_AI_MODEL

Модель Gemini для верстки и глоссария.

Рекомендуемое значение:

```text
LAYOUT_AI_MODEL=gemini-2.5-flash-lite
```

Если не знаете, что выбрать, не трогайте. В коде уже стоит это значение по
умолчанию.

### LAYOUT_AI_TIMEOUT

Сколько секунд ждать ответ Gemini для верстки.

Обычное значение:

```text
LAYOUT_AI_TIMEOUT=12
```

Если в логах часто появляется `Gemini-проверка верстки недоступна`, можно
увеличить до `18` или `25`.

### LAYOUT_AI_MAX_CHARS

Максимальная длина текста, который отправляется на Gemini-проверку верстки.

Обычное значение:

```text
LAYOUT_AI_MAX_CHARS=3200
```

Если текст длиннее, бот пропустит AI-верстку и применит локальные правила.

## Два отдельных процесса редактуры

В боте специально разделены две разные задачи.

### 1. Глоссарий: имена и термины

Функция: `edit_gemini()`.

Задача: исправить только плохие написания из `glossary.py`.

Пример:

```text
Кендри Паес -> Кендри Паэс
Кендри Кавидес -> Кендри Паэс
Ламин Ямал -> Ламин Ямаль
```

Что запрещено:

1. переписывать пост целиком;
2. добавлять новые вопросы;
3. добавлять новые ответы;
4. добавлять цитаты;
5. добавлять имена, которых не было в тексте;
6. добавлять факты из головы.

Чтобы Gemini не галлюцинировал, он больше не получает задачу `верни готовый
пост`. Он должен вернуть только JSON со списком точечных замен:

```json
{"replacements":[{"from":"Кендри Паес","to":"Кендри Паэс"}]}
```

Код применит замену только если фрагмент из `from` буквально есть в исходном
тексте. Если Gemini попытается вернуть новый пост про другого игрока, такая
правка не пройдет.

### 2. Литературная редактура русского текста

Функция: `edit_style_gemini()`.

Задача: сделать русский текст естественным, но не менять смысл.

Пример плохого текста:

```text
Новый 10-й Эквадора
```

Нормальный вариант:

```text
Новая «десятка» Эквадора
```

Пример плохого текста:

```text
В возрасте Кендри Паэс уже достиг большего успеха
```

Нормальный вариант:

```text
В свои годы Кендри Паэс уже достиг большего успеха
```

Литературная редактура тоже не имеет права возвращать пост целиком. Она тоже
возвращает только список маленьких замен:

```json
{"replacements":[{"from":"новый 10-й","to":"новая «десятка»"}]}
```

Код проверяет, что после замены:

1. не появились новые имена;
2. не появились новые числа;
3. не появились новые ссылки;
4. не появились новые факты;
5. текст не стал подозрительно длиннее;
6. не исчезли имена из глоссария.

## Настройки литературной редактуры

### STYLE_EDIT_ENABLED

Включает или выключает литературную редактуру Gemini.

Рекомендуемое значение:

```text
STYLE_EDIT_ENABLED=1
```

Выключить:

```text
STYLE_EDIT_ENABLED=0
```

Когда выключать:

1. если Gemini начал часто ошибаться;
2. если надо временно публиковать только DeepL + локальные правила;
3. если закончились лимиты Gemini.

### STYLE_EDIT_MODEL

Модель для литературной редактуры.

Если переменная пустая, используется `LAYOUT_AI_MODEL`.

Обычное значение:

```text
STYLE_EDIT_MODEL=gemini-2.5-flash-lite
```

### STYLE_EDIT_MAX_CHARS

Максимальная длина текста для литературной редактуры.

Рекомендуемое значение:

```text
STYLE_EDIT_MAX_CHARS=1800
```

Если поставить слишком большое значение, модель чаще будет ошибаться на длинных
постах. Для безопасности лучше держать 1500-2200.

### STYLE_EDIT_TIMEOUT

Сколько секунд ждать ответ Gemini для литературной редактуры.

Рекомендуемое значение:

```text
STYLE_EDIT_TIMEOUT=18
```

## Настройки OpenRouter

### OPENROUTER_KEY

Ключ OpenRouter.

Он нужен для:

1. AI-проверки смысловых дублей;
2. RAG-проверки Telegram-верстки.

Если ключа нет, бот не остановится. Он просто напишет в логах:

```text
AI-проверка дублей недоступна
RAG-проверка верстки пропущена: OPENROUTER_KEY не задан
```

### OPENROUTER_MODEL

Модель для AI-проверки дублей.

Пример:

```text
OPENROUTER_MODEL=meta-llama/llama-3.3-70b-instruct:free
```

Если бесплатная модель часто дает `429 Too Many Requests`, это значит, что
OpenRouter временно не дает новые запросы. Бот продолжит работу, но AI-дубли
будут проверяться хуже.

### FORMAT_RAG_ENABLED

Включает RAG-проверку верстки.

```text
FORMAT_RAG_ENABLED=1
```

Выключить:

```text
FORMAT_RAG_ENABLED=0
```

RAG не переводит текст и не редактирует смысл. Он только помогает с версткой:
переносы, цитаты, blockquote, пустые строки.

## Как редактировать glossary.py

Откройте `glossary.py`.

Каждая запись выглядит так:

```python
{
    "bad": "Кендри Паес",
    "good": "Кендри Паэс",
    "comment": "Эквадорский футболист",
},
```

Где:

`bad` - неправильный вариант, который часто появляется после перевода.

`good` - правильный вариант.

`comment` - пояснение для человека и модели.

Правила:

1. Не добавляйте в `good` целые предложения.
2. Не используйте глоссарий для литературной редактуры.
3. В глоссарии должны быть только имена, фамилии, клубы, турниры и устойчивые термины.
4. Если DeepL пишет фамилию в нескольких плохих вариантах, добавьте несколько записей.

Хорошо:

```python
{
    "bad": "Паес",
    "good": "Паэс",
    "comment": "Фамилия Kendry Paez",
},
```

Плохо:

```python
{
    "bad": "новый 10-й",
    "good": "новая десятка сборной Эквадора, которая должна стать лидером команды",
    "comment": "",
},
```

Почему плохо: это уже не термин, а переписывание смысла.

## Как настроить шаг 1: Twitter/X по ключевым словам

Шаг 1 ищет посты по словам.

Главные переменные:

```text
STEP1_KEYWORDS=kendry páez,kendry paez,kendrypaez
STEP1_ACCOUNTS=FabrizioRomano,TheChelseaForum
STEP1_LIMIT=50
SCWEET_AUTH_TOKEN=ваш_auth_token
```

`STEP1_KEYWORDS` - слова, по которым искать.

`STEP1_ACCOUNTS` - аккаунты, внутри которых искать. Если оставить пустым, поиск
будет шире.

`STEP1_LIMIT` - сколько твитов максимум смотреть за запуск.

`SCWEET_AUTH_TOKEN` - токен для Scweet. Без него шаг 1 и шаг 3 через Scweet не
работают.

## Как настроить шаг 2: Instagram

Шаг 2 берет посты Instagram через Apify.

Главные переменные:

```text
APIFY_TOKEN=ваш_apify_token
STEP2_INSTAGRAM_ACCOUNTS=chelseafc,kendrypaez
APIFY_INSTAGRAM_RESULTS_LIMIT=10
```

`APIFY_TOKEN` - ключ Apify.

`STEP2_INSTAGRAM_ACCOUNTS` - Instagram-аккаунты через запятую, без `@`.

`APIFY_INSTAGRAM_RESULTS_LIMIT` - сколько постов брать.

## Как настроить шаг 3: Twitter/X аккаунты

Шаг 3 смотрит свежие посты конкретных аккаунтов.

Главные переменные:

```text
STEP3_ACCOUNTS=FabrizioRomano,DiarioOleEc
STEP3_LIMIT=20
STEP3_SCAN_LIMIT=50
STEP3_NITTER_FALLBACK=1
```

`STEP3_ACCOUNTS` - список аккаунтов без `@`.

`STEP3_LIMIT` - сколько новых постов можно опубликовать.

`STEP3_SCAN_LIMIT` - сколько постов просмотреть, чтобы найти новые.

`STEP3_NITTER_FALLBACK=1` - если Scweet дал старые данные, бот попробует Nitter.

## Как настроить шаг 4: Instagram по хэштегам

Шаг 4 ищет Instagram-посты по хэштегам.

Главные переменные:

```text
STEP4_HASHTAGS=kendrypaez,chelsea
STEP4_ALLOWED_ACCOUNTS=chelseafc,kendrypaez
STEP4_SIGNATURE=#KendryPaez | Follow us
```

`STEP4_HASHTAGS` - хэштеги без `#`.

`STEP4_ALLOWED_ACCOUNTS` - если заполнено, бот публикует посты только от этих
аккаунтов. Если пусто, разрешены все аккаунты.

`STEP4_SIGNATURE` - подпись именно для шага 4.

## Подпись канала

Обычная подпись собирается из трех переменных:

```text
SIGNATURE_HASHTAG=KendryPaez
SIGNATURE_URL=https://t.me/your_channel
SIGNATURE_URL_TEXT=Follow us
```

В посте получится:

```text
#KendryPaez | Follow us
```

`Follow us` будет ссылкой.

Откройте GitHub:

1. Репозиторий.
2. Вкладка `Actions`.
3. Нужный запуск.
4. Job `run`.

Полезные строки:

```text

```

```text

```

Проблема с Gemini на этапе глоссария. Проверьте `GEMINI_API_KEY`.

```text
Gemini глоссарий отклонён: нет безопасных точечных замен
```

Gemini предложил небезопасную замену. Бот её не применил.

```text
Gemini редактура текста поправлена
```

Литературная редактура сработала.

```text
Gemini редактура текста отклонена: нет безопасных точечных замен
```

Gemini предложил слишком рискованную правку. Бот оставил исходный текст.

```text
Gemini-верстка поправлена
```

Gemini поправил только Telegram-верстку.

```text
RAG-проверка верстки недоступна: неверный ответ модели
```

OpenRouter/RAG вернул не тот формат. Это не критично: бот продолжит работу.

```text
AI-проверка дублей недоступна: 429
```

OpenRouter временно ограничил запросы. Это не ошибка кода.

## Что делать, если плохой перевод прошел в канал

1. Скопируйте плохой текст.
2. Найдите, что именно не так.
3. Если ошибка в имени или термине, добавьте запись в `glossary.py`.
4. Если ошибка в естественности русского языка, не добавляйте это в глоссарий.
5. Для частых ошибок DeepL лучше добавить локальное правило в `apply_local_translation_fixes()` в `bot.py`.
6. Если Gemini предложил опасную правку, проверьте `format_model/hard_cases.jsonl`.

Пример:

```text
кендри паэс
```

Это имя с маленькой буквы. Исправляется локальными правилами и нормализацией
капитализации.

Пример:

```text
новый 10-й Эквадора
```

Это не глоссарий. Это литературная правка. Исправляется локальным правилом или
этапом `STYLE_EDIT_ENABLED=1`.

Пример:

```text
Кендри Кавидес
```

Это ошибка имени. Добавляется в `glossary.py` как плохая форма для `Кендри Паэс`.

## Рекомендуемые настройки качества

Для безопасного режима:

```text
GEMINI_API_KEY=ваш_ключ
LAYOUT_AI_MODEL=gemini-2.5-flash-lite
STYLE_EDIT_ENABLED=1
STYLE_EDIT_MODEL=gemini-2.5-flash-lite
STYLE_EDIT_MAX_CHARS=1800
STYLE_EDIT_TIMEOUT=18
FORMAT_RAG_ENABLED=1
```

Для максимально осторожного режима, если Gemini снова ведет себя плохо:

```text
STYLE_EDIT_ENABLED=0
FORMAT_RAG_ENABLED=0
```

При этом глоссарные локальные замены всё равно останутся.

## Частые проблемы

### Бот ничего не публикует

Проверьте:

1. есть ли новые посты;
2. не слишком ли старые посты;
3. есть ли медиа;
4. не опубликован ли этот пост раньше;
5. правильно ли задан `CHANNEL_ID`;
6. является ли бот админом канала.

### В логах 403 при скачивании видео

Это значит, что Twitter/X не дал скачать видео. Бот пропустит этот пост, если
не сможет скачать ни одно медиа.

### В логах 429 OpenRouter

Это лимит OpenRouter. Можно:

1. подождать;
2. поменять модель;
3. пополнить баланс;
4. временно выключить RAG: `FORMAT_RAG_ENABLED=0`.

### Gemini что-то предложил, но бот отклонил

Это нормально. Лучше пропустить сомнительную правку, чем опубликовать выдуманный
текст.

Проверяйте `format_model/hard_cases.jsonl`: там сохраняются опасные варианты.

## Главное правило

`glossary.py` - только для имен и терминов.

Литературная редактура - отдельно через `STYLE_EDIT_ENABLED`.

Если смешать эти задачи, модель начинает галлюцинировать. Поэтому бот разделяет
их на два этапа и принимает только маленькие точечные замены.
